The Post-Quantum Triad Hybrid Fusion

Our architecture utilizes a Defense-In-Depth strategy, wrapping AES-256-GCM payloads inside mathematically distinct protocols verified by NIST.

1. Lattice KEM

CRYSTALS-Kyber (ML-KEM) handles lightning-fast asymmetric key encapsulation to generate the primary key material.

2. Code-Based KEM

3. Hash Signatures

Deep Dive: Kyber-1024 Module Learning With Errors

Kyber operates on the hard mathematical problem of finding short vectors in a lattice (MLWE). Even an omnipotent quantum computer cannot solve this efficiently.

  • Performance: Encapsulation takes <1ms.
  • Security Level: NIST Level 5 (equivalent to AES-256 brute force difficulty).
  • Key Sizes: 1568 bytes (Public Key), 3168 bytes (Secret Key).

The MLWE Problem

Given a public matrix A and a vector b = As + e (where s is the secret key and e is small random noise), finding s is computationally intractable.

Deep Dive: Classic McEliece Goppa Codes

Unlike Kyber, McEliece relies on the hardness of decoding a general linear code. Developed in 1978, it has survived literally decades of intense cryptanalysis.

  • Tradeoff: Massive public keys (~1 Megabyte) but minuscule ciphertexts and incredibly fast decapsulation.
  • Role: Acts as the "failsafe" layer in case lattice-based cryptography is eventually broken mathematically.

Implementation Note

Browser chunking enables us to shuttle megabyte-sized McEliece-8192128 public keys across WebSocket payloads without locking the JavaScript main thread or exhausting stack sizes.

Deep Dive: SPHINCS+ Stateless Signatures

SPHINCS+ (SLH-DSA) is built entirely from secure hash functions (SHA-256). It does not rely on algebraic problems like factoring (RSA) or discrete logarithms (ECC).

// Signature Generation Flow
1. Hash message M with randomized prefix
2. Sign root of FORS (subset-resilient tree)
3. Sign path up via WOTS+ (one-time sig)
4. Authenticate through hyper-tree layers
5. Output ~49KB stateless signature

Why Stateless? Stateful hash signatures (like XMSS) require tracking which keys have been used. Re-using a key breaks security. SPHINCS+ solves this mathematically via randomized hyper-tree indexing, eliminating the risk of catastrophic state-sync failures in real-time chat.

Zero-Trust Media Transport Web Crypto API

Our platform handles Text, Audio, Documents, and Images. The server never observes plaintext data, relying exclusively on client-side memory buffer encryption.

  • AES-256-GCM: Payload encryption executes entirely within the browser sandbox using the native high-performance Web Crypto API.
  • Blind Backend: The FastAPI server acts purely as a routing relay. It receives Base64 encrypted ciphertexts alongside SPHINCS+ signatures.
  • Blob Decoding: The receiving browser verifies the signature, applies the HKDF AES key to the byte array, and constructs media chunks natively (URL.createObjectURL).

Asymmetric Key Encapsulation Engine

During the `Messenger` P2P phase, how do users communicate asynchronously?

Alice pulls Bob's persistent Kyber+McEliece Public Keys from the server, generates a 32-byte secret `S`, encapsulates it, and sends the resulting ciphertexts to Bob. Bob uses his secret key to decapsulate back to `S`, forming a perfectly secure E2E tunnel.

1. S = os.urandom(32)
2. CT_A, SS_A = Kyber.encap(PK_Alice)
3. CT_B, SS_B = Kyber.encap(PK_Bob)
   (Same for McEliece)
4. Send CT_B to Bob over WebSocket
5. Bob decapsulates -> S
6. Alice & Bob use S for AES-GCM
